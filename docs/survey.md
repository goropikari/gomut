# Go言語 ミューテーションテースティングツール開発に向けた機能調査報告書

Go言語における主要なミューテーションテースティングツールである **go-gremlins/gremlins** と **avito-tech/go-mutesting** の調査・比較を行い、新規ツール開発に向けた統合機能要件をまとめました。

---

## 1. 2大ツールの特徴・アプローチ比較

| 評価項目                  | go-gremlins/gremlins                                                              | avito-tech/go-mutesting                                                                       |
| :------------------------ | :-------------------------------------------------------------------------------- | :-------------------------------------------------------------------------------------------- |
| **開発思想**              | **「実行速度」と「開発体験（DX）」の重視**<br>実用的な時間でのCI/CD連携を目指す。 | **「網羅性」と「AST操作の柔軟性」の重視**<br>コードのあらゆる可能性に対して変異を試みる。     |
| **変異対象の選定**        | **カバレッジベース（高速）**<br>テストで実行されたコードのみに変異を仕込む。      | **ファイル/パッケージ全体**<br>テストの有無に関わらず、指定されたコードすべてに変異を仕込む。 |
| **変更差分(PR)実行**      | **対応** (`diff` サブコマンドでGitの差分のみ実行可能)                             | **非対応** (基本はファイルまたはパッケージ単位の実行)                                         |
| **変異（Mutator）の種類** | 主要な5〜6ジャンルに厳選（実用性重視）                                            | 非常に豊富（ビット演算、インクリメント、代入反転など）                                        |
| **並行実行サポート**      | 対応（複数ワーカーによる並行テスト実行）                                          | 対応（`-exec` を用いた並行処理制御）                                                          |
| **設定方法**              | YAMLファイル ＋ CLIフラグ                                                         | CLIフラグ中心                                                                                 |

## 2. 各ツールの強みと弱み（自作ツールへの教訓）

### go-gremlins/gremlins

- **強み:**
  - 事前に `go test -coverprofile` を実行し、**テストがカバーしていない箇所の変異をスキップする**ため圧倒的に高速。
  - Gitとの連携（Diff実行）が標準で組み込まれており、プルリクエスト（PR）のCIに組み込みやすい。
- **弱み:**
  - 変異パターン（Mutator）の種類が限定的で、より複雑な論理バグのシミュレーションには物足りない場合がある。

### avito-tech/go-mutesting

- **強み:**
  - Goの `go/ast` (抽象構文木) のパースと書き換えロジックが非常に洗練されている。
  - 変異パターンが豊富。ビット演算子（`&` ⇔ `|`）や、`+=` ⇔ `-=`、インクリメント（`++` ⇔ `--`）など、細かなバグも再現できる。
- **弱み:**
  - 未カバーのコードにも愚直に変異を仕込んでテストを走らせるため、大規模プロジェクトでは実行時間が膨大になる。
  - CI/CDフレンドリーな機能（差分実行や構造化レポート）が標準では弱い。

## 3. 統合・自作ツールの機能要件マップ

両ツールの「いいとこ取り」をした、**「高速かつ網羅的な最強のGoミューテーションツール」**の設計に必要な要件です。

### ① コード解析・変異生成（Mutator）モジュール

`go-mutesting` の網羅的な変異ロジックをベースに実装します。

- **算術演算子の反転:** `+` ⇔ `-`、`*` ⇔ `/`、`%` ⇔ `*`
- **比較演算子の変更:** `<` ⇔ `<=`, `>` ⇔ `>=`, `==` ⇔ `!=`
- **論理演算子の反転:** `&&` ⇔ `||`
- **制御フローの変更:** `break` ⇔ `continue`、`if` 条件の否定（`!` の付与・除去）
- **代入・ビット演算の変異:** `+=` ⇔ `-=`、`&` ⇔ `|`
- **早期リターン（ガード句）の破壊:** `if err != nil { return err }` の中身を空（`nil`）にする、または早期リターン自体を削除

### ② 実行最適化・高速化モジュール

`gremlins` の高速化手法を必須要件として組み込みます。

- **カバレッジ駆動型ミューテーション:** 事前にカバレッジを取得し、未テストのコード行への変異を自動スキップする。
- **テストケースの局所化:** 変異を仕込んだパッケージ（または影響範囲のあるパッケージ）のテストのみを実行する。
- **マルチコア並行実行:** `go build` と `go test` を独立した一時ディレクトリ（`GOTMPDIR` 等）で並行実行するワーカープール機構。
- **無限ループ対策（タイムアウト制限）:** 変異によって条件式が変わり、テストが無限ループに陥った場合、通常のテスト実行時間の数倍（例: 2〜5秒）で強制終了（`TIMED OUT`）させる。

### ③ CI/CD & 開発体験（DX）モジュール

モダンな開発パイプラインに適合させるための機能です。

- **Git Diff 実行機能:** `git diff origin/main` などから変更されたファイル・行数を特定し、その周辺のみに変異を限定する。
- **クオリティゲート（品質閾値）:** 「ミューテーションスコア（バグを検知できた割合）が80%未満なら終了コード `1` で落とす」といったCI連携機能。
- **除外設定（Ignore設定）:** YAML/TOMLファイルで、自動生成コード（`*.pb.go`）やモックファイル（`*_mock.go`）を評価対象から除外できる。

### ④ レポートモジュール

結果を開発者に分かりやすく伝えるための出力機能です。

- **ミュータント状態の厳密な分類:**
  - `KILLED`: テストが落ちた（バグを正しく検知できた ✅）
  - `LIVED`: テストが通ってしまった（テストの考慮漏れ ❌）
  - `NOT COVERED`: そもそもテストが通っていない場所（スキップ対象）
  - `TIMED OUT`: 無限ループ等でタイムアウトした（検知成功扱いとすることが多い）
  - `NOT VIABLE`: 変異後のコードがコンパイルエラーになった（無効なミュータント）
- **表示フィルタリング:** CLIの標準出力には `LIVED`（生き残った不具合候補）のみを表示し、詳細な全結果は JSON / HTML 形式でファイル出力する。
