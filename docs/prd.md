# PRD: mutation 実行の高速化

## 1. タイトル

mutation 実行の高速化

## 2. 要約

`gomut test` の mutation 実行時間を短縮する。既存の並列数の上限を守りながら、実行エンジン内部の不要な処理を削減する。mutation ごとの `KILLED` / `LIVED` などの判定と、JSONL の 1 mutation 1 record を維持する。

## 3. 課題

現在の mutation testing は候補ごとに `go test` を実行するため、候補数に比例して実行時間が増加する。並列実行は利用できるが、ワーカーごとに隔離用のリポジトリコピーを作成するなど、テストそのもの以外の処理も実行時間とディスク I/O を増加させる。

`gomut test ./...` と単一パッケージを対象にした実行を、結果の信頼性を損なわずに高速化する必要がある。

## 4. 目的

同一環境・同一設定で実行した既存実装と比べ、`sample` ディレクトリを対象とする mutation 実行の総所要時間を 50% 以上短縮する。

## 5. 非対象

- mutation 種別の追加・削除
- `KILLED`、`LIVED`、`NOT_COVERED`、`TIMED_OUT`、`NOT_VIABLE` の意味変更
- mutation ごとの判定を省略する近似的な結果判定
- JSONL における mutation ごとの record 出力の廃止
- ユーザーが指定する `--parallel` の上限を超える CPU 並列度の導入
- CLI の新しい速度優先オプションや設定項目の追加
- `sample` 以外の外部リポジトリを性能目標の合否基準にすること

## 6. 対象ユーザー

- ローカル開発時または CI で `gomut test` を実行する Go 開発者
- 単一パッケージまたは `./...` を対象に mutation testing を実行する利用者

## 7. スコープ

### 対象

- `gomut test <単一パッケージ>` の mutation 実行エンジン
- `gomut test ./...` の mutation 実行エンジン
- 隔離用ワークスペースの準備・再利用・後始末
- `go test` 起動前後の不要な処理、キャッシュ利用可能性、ディスク I/O の最適化
- `sample` を対象とした再現可能な性能ベンチマークと比較手順

### 許容する変更

- mutation の内部スケジューリング
- JSONL の record 出力順
- stderr の進捗・補助メッセージ
- 既存 CLI の外部契約を満たす範囲での内部実行方式

## 8. ユーザーフロー

1. 利用者が `gomut test ./sample` または `gomut test ./...` を実行する。
2. gomut が baseline test、候補検出、対象絞り込みを実行する。
3. gomut が設定済みの `--parallel` 値を上限として mutation を実行する。
4. gomut が各 mutation の結果を分類し、JSONL 有効時は mutation ごとに 1 record を出力する。
5. gomut が stderr に集計と補助情報を出力する。

## 9. 機能要件

1. 同一入力・同一実行設定では、最適化前後で各 mutation の最終結果分類が一致すること。
2. JSONL 出力時は、すべての対象 mutation について 1 mutation につき 1 record を出力すること。
3. 各 JSONL record は既存 Output Contract の必須 mutation 情報と対象・集計情報を保持すること。
4. `--parallel` が明示された場合、その値を超える同時 mutation 実行を行わないこと。
5. `--parallel` 未指定時も、既存のデフォルト並列数の意味を変更しないこと。
6. mutation 実行中のキャンセル、タイムアウト、隔離環境の後始末が従来どおり機能すること。
7. `sample` を対象とした性能計測を自動実行または文書化された手順で再現可能にすること。
8. 性能比較では、同じ Go バージョン、マシン、`--parallel` 設定、キャッシュ条件を使用し、複数回の計測結果を記録すること。

## 10. 制約と依存関係

- mutation 実行前に baseline の `go test` が成功すること。
- Go 1.26 以上を前提とする。
- 実行環境で `go test` が利用可能であること。
- 既存の作業ツリーを mutation により変更しない隔離性を維持すること。
- stdout は JSON Lines の出力先、集計・補助メッセージは stderr という Output Contract を維持すること。
- 実装は CLI 表面ではなく、実行エンジン内部の最適化に限定すること。

## 11. 成功指標

- `sample` を対象にした mutation 実行の総所要時間が、基準実装比で 50% 以上短縮されること。
- 比較対象は `gomut test ./sample` と `gomut test ./...` の両方とする。
- 最適化前後で mutation ごとの結果分類と JSONL record 数が一致すること。
- 指定した `--parallel` を超える同時 mutation 実行が発生しないこと。

## 12. リスクと例外

- Go build/test cache の状態で計測値が大きく変動する。計測手順でキャッシュ条件を固定または明記する。
- ワークスペース再利用は mutation 間の状態漏れを起こす可能性がある。各 mutation のソース状態と生成物を独立させる検証が必要である。
- 並列度を上げずに高速化するため、ボトルネックの計測なしに最適化すると改善が限定的になる。実装前後でプロファイルまたは所要時間の内訳を確認する。
- 出力順を変更できるため、順序に依存する利用者がいる可能性がある。変更する場合はリリースノートで明示する。
- `./...` の対象範囲や外部依存によって改善率が変わる。50% の合否は `sample` を基準に判定する。

## 13. リリース計画

1. 現行実装で `sample` の基準計測と結果分類のスナップショットを取得する。
2. 実行エンジン内部を最適化し、隔離性・キャンセル・タイムアウトの自動テストを追加または更新する。
3. `sample` に対し単一パッケージと `./...` の性能・結果同等性を検証する。
4. `go test ./...`、`make fmt`、`make lint` を実行する。
5. 50% 短縮と互換性要件を満たした後にリリースする。JSONL 出力順を変更した場合は変更点として告知する。

## 14. 未解決事項

- 性能測定に用いる反復回数、統計値（中央値・最小値など）、キャッシュのウォームアップ手順。
- `sample` の候補数が将来も性能差を安定して検知できるか。
- 計測コマンドを Makefile に追加するか、専用の Go benchmark にするか。

## 15. 前提

- `sample` は現時点で既存の単一パッケージおよび `./...` 実行の性能比較に利用できる。
- 高速化は内部実装で達成し、利用者は新しいフラグを指定しない。
- JSONL record の順序は外部互換性の必須要件ではない。

## 16. 付録

### 比較対象コマンド

```bash
gomut test ./sample
gomut test ./...
```

### 受け入れ条件

- 同一条件の `sample` ベンチマークで、両対象コマンドの総所要時間が基準実装比 50% 以上短縮される。
- 各 mutation の結果分類と JSONL record 数が基準実装と一致する。
- `--parallel` の指定値を超えて mutation を同時実行しない。
- 既存作業ツリーが mutation 実行で変更されない。
